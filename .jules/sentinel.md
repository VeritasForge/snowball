## 2025-05-15 - IDOR in Repository Layer
**Vulnerability:** The `AccountRepository.list_all()` method returned all accounts in the database regardless of ownership, and this was directly exposed via the `GET /accounts` endpoint.
**Learning:** The initial repository design focused on "listing entities" rather than "listing user entities", and the API layer failed to enforce the ownership constraint, assuming the repository would handle filtering or that the list operation was generic.
**Prevention:** Always implement `list_by_owner` or similar methods in repositories for user-owned resources. Ensure API endpoints explicitly pass the `current_user` ID to data access layers.

## 2024-03-27 - [CRITICAL] Widespread IDOR in Resource Mutation Endpoints
**Vulnerability:** Found multiple API routes (`update_account`, `delete_account`, `create_asset`, `update_asset`, `delete_asset`, `execute_trade`) vulnerable to Insecure Direct Object Reference (IDOR). These routes did not verify that the authenticated `current_user` was the actual owner of the account or the parent account of an asset before performing mutations, allowing an attacker to modify or delete any user's data.
**Learning:** The FastAPI dependency injection logic for extracting `current_user` was missing on most endpoints, except for the `list_accounts` route and `create_account` route. Relying only on token authentication is insufficient; the authorization of the specific resource being accessed must be verified, especially for associated entities (e.g. updating an asset requires checking if the asset's account belongs to the user).
**Prevention:** Every endpoint modifying, accessing, or deleting a specific resource must include the `current_user` dependency and an explicit ownership check (`if resource.user_id != current_user.id: raise HTTPException(status_code=403)`). For nested resources like `Asset`, verify ownership by fetching the parent `Account` and checking its `user_id`.
